## 1.5 多数据集机制(扩展数据集)

VGP-Nav 支持多个数据集,**加新数据集只需在 `vgpnav/datasets.py` 增加一条 `DatasetSpec`,不改任何脚本/核心代码**。

### 1.5.1 怎么切换数据集
所有脚本里 `cfg = Config()` 默认读环境变量 `VGPNAV_DATASET`(没有则 `Mapping_C8`):
```bash
export VGPNAV_DATASET=ChuangfuTower_floor1     # 之后所有脚本都对创富大厦1楼生效
# 或单条命令前缀: VGPNAV_DATASET=ChuangfuTower_floor1 $PY scripts/build_database.py
```
产物自动隔离到 `outputs/{数据集名}/`(db/query/web 互不干扰)。也可在代码里 `Config(dataset="名字")` 显式指定。

### 1.5.2 已注册数据集(`vgpnav/datasets.py`)
| 数据集 | 相机模型 | 相机高度 | 4环视布局 (camera_1/2/3/4) | 数据 / 帧数 | IMU | 说明 |
|---|---|---|---|---|---|---|
| `Mapping_C8` | MEI(`params.yaml`) | 1.3m | 前左/前右/后右/后左 | `data/Mapping_C8`, 626帧, `map_stride=1` | 无 | 深港国际C8(DEEPROUTE 一层),小车环视 |
| `ChuangfuTower_floor1` | KB4(`camera_param.json`) | 1.67m | 左后/左前/右前/右后 | `data/ChuangfuTower_floor1`, 3215帧→`map_stride=5`→约643帧 | 有 | 创富大厦 1 楼,头盔环视,室外 |
| `ChuangfuTower_floor28` | KB4(`camera_param.json`) | 1.67m | 左后/左前/右前/右后 | `data/ChuangfuTower_floor28`, 180帧, `map_stride=1` | 有 | 创富大厦 28 楼,头盔环视,室内办公 |

### 1.5.3 加一个新数据集(3 步)
1. **准备图片**:4路环视命名 `{timestamp}_camera_{1..4}.jpg`,放到某目录(如 `data/新数据集名`);KB4 鱼眼相机还需把 `camera_param.json` 放进该目录。
2. **加 spec**:在 `vgpnav/datasets.py` 的 `DATASETS` 增加一条(字段 name/data_dir/camera_height_m/cam_layout/cam_params/map_stride/n_query):
   ```python
   "新数据集名": DatasetSpec(
       name="新数据集名",
       data_dir="/abs/path/新数据集名",
       camera_height_m=1.67,          # 相机离地高度(米) —— 尺度锚, 最关键参数
       cam_layout=[("左前",2),("右前",3),("左后",1),("右后",4)],  # 4环视2x2: 左上,右上,左下,右下(前进方向在上排)
       cam_params="/abs/path/新数据集名/camera_param.json",  # KB4鱼眼: data_dir 内的 camera_param.json; MEI: 指向 params.yaml; ""=复用 C8 的 MEI params.yaml
       map_stride=5, n_query=25),     # 长序列调大 stride 抽样
   ```
3. **跑流程**:`export VGPNAV_DATASET=新数据集名` 后照 §0 / §3 跑(建图)、§4(导航测试)。

**关键说明**:
- `camera_height_m`:相机光学中心离地高度。机器人/头盔不同就改此值(尺度由它线性锚定,影响整图度量)。
  创富大厦是头戴头盔,主锚 camera_1 离地 ≈ 身高170cm − 头顶到头盔帽沿~12cm + 相机在帽沿上9.5cm ≈ 1.67m。
- `cam_layout`:决定 web 4环视 2×2 怎么摆(前进方向恒在上排)。按该数据集 4 相机的物理方位填,`index.html` 会动态生成。
- `cam_params` 与相机模型:**C8 是 MEI 鱼眼**,内参 `[xi,fx,fy,cx,cy]+[k1,k2,p1,p2]`,用 `memory-nav/cam/params.yaml`;**创富两层是 KB4 鱼眼**(ISX031,1920×1536),内参 `efl/cod/k1..4`,各自 `data_dir` 内的 `camera_param.json`。模型按 `cam_params` 文件类型区分(`.yaml`=MEI,`.json`=KB4),给错会去畸变错。
- `map_stride`:轨迹帧抽样步长,长序列调大(floor1 用 5,间距约 0.40m)。`n_query`:held-out 评测帧数。
- ⚠️ **短序列**(如 floor28 180帧):VGGT 滑窗链式重建尺度漂移大、常无回环(PGO 只能 z 拉平、修不了 x-y),建图质量受限——**数据越长越完整越好**。

### 1.5.4 数据集特定检索参数(floor28 强混叠 / floor1 稀疏DB)
**背景**:floor28 是开放办公区,工位高度重复造成强感知混叠。诊断显示 VPR top1 命中真值很准(10/10 <3m),但默认检索的多样性选择会把远处相似工位选作参考、污染运动平均,定位中位 5.40m。

**解决**:在 `vgpnav/datasets.py` 的 `DatasetSpec` 增加 `retrieval_radius_m` / `anchor_topm` / `k_refs` 三个覆盖字段(默认 0 = 用 `config.py` 全局默认 7.0 / 5 / 10)。floor28 设:
```python
retrieval_radius_m=3.0,   # 检索半径(米), 小半径只取近邻
anchor_topm=1,            # 以 VPR best 为锚(取 top1)
k_refs=5,                 # 少 refs, 避免远处相似工位污染运动平均
db_sub=1,                 # 密 DB: query held-out、其余 170 帧全做 DB, 最近参考更贴 query
```
即“以 VPR best 为锚、小半径、少 refs”,定位中位 5.40→2.25m;**再加 `db_sub=1` 密 DB**(query held-out、其余 170 帧全做 DB, DB 间距 4.4→1.8m, 让检索到的最近参考更贴 query) → 2.25→1.09m(相对 1.23%, **总评良好**)。覆盖在 `vgpnav/config.py` 的 `__post_init__` 里实现:`self.retrieval_radius_m = spec.retrieval_radius_m or self.retrieval_radius_m`(字段为 0 时回退全局默认)。

**floor1** 是另一类问题:DB 稀疏(节点间距 12.8m),默认半径 7m 比间距还小、refs 不足会补到几十米外的相似街角,
定位中位 12.61m。设 `retrieval_radius_m=15.0, anchor_topm=1, k_refs=8`(半径 >DB 间距)后定位中位降到 6.14m(相对 0.96%)。

**只有 C8(室内中等场景、结构差异充分)沿用全局默认(7.0 / 5 / 10)。**

## 3. 建图流程逐步详解(原始数据 → 最终地图)

> 表格速览,逐条详见下文。"耗时"为 L40 参考值。

| # | 命令 | 作用 | 主要输入 | 主要输出 | 耗时 |
|---|---|---|---|---|---|
| ① | `build_database.py` | VGGT滑窗建库+重力对齐+地面锚定 | 4路图 + cam params (+imu.csv) | trajectory/db/query/global_points.npz, meta.json | ~5-10min |
| ② | `compute_4cam_desc.py` | 4路环视VPR描述子 | 4路图 | allframe_desc_4cam.npy | ~3min |
| ③ | `pgo_fix.py --sim 0.70 --dmax 4 --simfar 0.85 --max_nfev 4000` | 分层回环PGO | trajectory_orig/trajectory + 4cam desc | trajectory_pgo.npz, pgo_before_after.png | ~1-15min* |
| ④ | `apply_pgo.py` | 写回校正位姿 | trajectory_pgo.npz | 覆盖 trajectory/db/query;备份 trajectory_orig | <1s |
| ⑤ | `build_4cam_points.py` | 360°融合点云重建 | trajectory(校正)+4路图 | global_points_4cam.npz | ~10-15min |
| ⑥ | `export_cams.py` | 4路环视缩略图 | db.npz + 4路图 | web/thumbs/n*_c*.jpg | ~2-3min |
| ⑦ | `run_query.py` | 定位评测+局部占据+A* | db/query/global_points + query 4路图 | results.npz + 4张评测图 | ~3-5min |
| ⑧ | `export_web.py` | 导出最终web | trajectory/4cam点云/db/query/results/node_names | web/{index.html,data.js} | ~1-2min |

\* PGO 耗时取决于 `max_nfev`(默认 800)。scipy 数值雅可比单线程较慢;大漂移真闭环需 `--max_nfev 4000`,会跑十几分钟(见 §7)。

### ① `build_database.py` —— 建库
- **做什么**:`build_database()` 执行 VGGT 滑窗链式重建(两遍尺度)+ 全局重力对齐(有 IMU 用加速度自标定重力方向,无 IMU 走视觉)+ 地面锚定(按 `camera_height_m`)+ DB/query 划分 + DB 描述子;然后画 `database_overview.png`(俯视轨迹+侧视相机高度验证)。
- **数据集相关参数**(来自 `vgpnav/datasets.py` 的 spec):`map_stride`(C8=1 全帧 / floor1=5 抽样)、`camera_height_m`(C8=1.3 / 创富=1.67)、`cam_params`、`n_query`。其余建图超参在 `vgpnav/config.py`(`db_sub`、`chunk_size`、`chunk_overlap`、`device`)。
- **输出**:`outputs/{数据集}/db/{trajectory,db,query,global_points}.npz` + `meta.json`。

### ② `compute_4cam_desc.py` —— 4路描述子
- **做什么**:对全部建图帧的 camera_1~4 去畸变后算 SelaVPR 描述子,存 `allframe_desc_4cam.npy (N_traj,4,D)`。
- **为何需要**:回环检测要识别"反向经过同一地点"(单目漏检),4路交叉相似度能抓到(详见原理文档 §3.2)。
- **依赖**:① 的 `trajectory.npz`(确定帧集)。

### ③ `pgo_fix.py` —— 分层回环 PGO
```bash
$PY scripts/pgo_fix.py --sim 0.70 --dmax 4 --simfar 0.85 --max_nfev 4000 --gap 40 --wloop 3.0
```
- **参数**(默认值见脚本 argparse):
  - `--sim`(默认 0.55,建图常用 0.70):**近距离**回环的 4 路交叉相似度阈值(局部重访)。
  - `--dmax`(默认 4):**近距离**回环的建图空间距离上限(米);此距离内 sim>`--sim` 即认回环。
  - `--simfar`(默认 0.85,★分层回环关键):**远距离**(超 dmax)回环的高相似阈值。近距离靠 sim+距离,远距离靠**高外观相似**区分:绕一圈回起点的**大漂移真闭环**外观极相似(>0.9)→ 捕获;**矩形对边**等几何巧合相似度中等 → 不触发。`simfar` 太小误连对边,太大漏真闭环。
  - `--max_nfev`(默认 800):PGO 最大迭代;把大漂移真闭环从几十米误差拉回需调大到数千(如 4000)才收敛。
  - `--gap 40`:回环最小帧间隔。`--wloop 3.0`:回环边权重。`--topk 4`:每帧最多保留回环数。
- **输入**:优先 `trajectory_orig.npz`(若存在),否则 `trajectory.npz`;+ `allframe_desc_4cam.npy`。
- **输出**:`trajectory_pgo.npz`(校正位姿,z 拉平到相机高度)+ `pgo_before_after.png`。
- ⚠️ **orig 优先**:`pgo_fix` 优先读 `trajectory_orig.npz`。**重新建库**后想重跑 PGO,先 `cp outputs/{数据集}/db/trajectory.npz outputs/{数据集}/db/trajectory_orig.npz` 让 PGO 基于当前数据(首次全新流程无此问题,因 orig 尚不存在)。

### ④ `apply_pgo.py` —— 写回
- **做什么**:把 `trajectory_pgo.npz` 校正位姿按 frame_idx 写回 `trajectory/db/query.npz`;**首次运行备份 `trajectory_orig.npz`**。
- **验证**:`apply` 后查同一物理点两趟经过的建图坐标是否对齐(回环修好则二者重合)。

### ⑤ `build_4cam_points.py` —— 360°融合点云
```bash
$PY scripts/build_4cam_points.py --stride 6   # --limit N 仅重建前N关键帧并出测试图
```
- **做什么**:用**对齐后**的 cam1 位姿为锚,每 `stride` 帧把 cam1 邻帧+cam2/3/4 送 VGGT,地面(按 `camera_height_m`)定尺度,融合成 `global_points_4cam.npz`(`points`+`kt`)。
- **依赖**:④ 写回的校正 `trajectory.npz`。这是**占据地图的点云来源**。

### ⑥ `export_cams.py` —— 4路环视缩略图
- **做什么**:为每个 DB 节点生成 cam1~4 去畸变缩略图 `web/thumbs/n{i}_c{c}.jpg`(C8 共 836 张)。web 的**4环视机器人视图/DB详情**靠这些文件。

### ⑦ `run_query.py` —— 定位评测(端到端 pipeline)
- **做什么**:对 held-out query(C8=25)跑完整 `VGPNav.run`(VPR→检索→VGGT→运动平均→尺度→局部占据→A*),输出定位误差 + 4 张图。
- **输出**:`outputs/{数据集}/query/results.npz`(定位误差,供 ⑧ 在 web 上展示)+ `localization_overview/query_montages/astar_demo/global_occupancy.png`。

### ⑧ `export_web.py` —— 导出最终 web(含最终占据地图)
- **做什么**:读 4路点云 + 轨迹 + DB + query + (可选)results/node_names,**生成最终灰度占据栅格**:栅格范围取轨迹包围盒+margin 并裁掉范围外离群点(避免 VGGT 玻璃/远景离群点撑爆地图),`grid_robust(min_hits=6, min_vext=0.3)` 滤噪,机器人走过的轨迹走廊(半径 0.75m)强制 free。连同轨迹/DB/定位结果写入 `data.js`;并把源 `web/nav.html` 拷成本数据集 `web/index.html`(导航页)、`web/portal.html` 拷成 `outputs/index.html`(多地图总入口)。
- **输出**:`outputs/{数据集}/web/{index.html, data.js}`(thumbs 由 ⑥ 提供)+ `outputs/index.html`(总入口)。

### 建图质量评估 —— `assess_quality.py`
建图 8 步跑完后,对地图质量做一次全面体检——**不重跑建图**,只读已有产物算指标,输出 `outputs/{数据集}/quality.json`(一套全面指标 + good/warn/bad 评分)。
```bash
VGPNAV_DATASET=<数据集> $PY scripts/assess_quality.py   # 评估单个数据集
$PY scripts/assess_quality.py                            # 不设环境变量则遍历全部 3 个数据集
```
- **输入**(均为已有产物,无需重跑建图):`db/`(trajectory.npz、db.npz、allframe_desc_4cam.npy、global_points_4cam.npz)、`query/results.npz`、`web/data.js` 的占据栅格。
- **五类指标**:
  - **A 覆盖度**:DB 节点数 / 轨迹长度 / 地图范围 / 平均 DB 间距 / 点云数。
  - **B 轨迹一致性**:相邻帧位移中位与 p95、突跳比、尺度符号。
  - **C 回环闭合**:视觉相似(4 路交叉描述子)的远时序帧对的建图空间距离——同地点应闭合(<2m)。
  - **D 定位精度**:平移误差中位、相对误差(=误差 / 场景对角线,跨尺度可比)、<1m / <2m 比例、旋转中位。
  - **E 占据连通**:free / 占据 / 未知占比、最大连通区占比、DB 可达率。
- **总评**:每类指标判 good/warn/bad,汇总成总评——**良好** / **一般**(有 warn 项) / **差**(有 bad 项)。
- **可视化**:总入口 `http://192.168.50.72:8199/` 的第 4 个 tab「质量评估」实时展示(A-E 卡片 + 评分 chips + 总评色条 + ▶运行评估 按钮)。

## 7. 注意事项与操作提示

1. **PGO 较慢**:`pgo_fix.py` 用 scipy `least_squares` + 数值雅可比 + 单线程,迭代慢。`--max_nfev` 默认 800;遇绕圈回起点的大漂移真闭环(误差几十米)需 `--max_nfev 4000` 才收敛,会跑十几分钟。
2. **重建库后重跑 PGO**:`pgo_fix` 优先读 `trajectory_orig.npz`(可能是上次建库的)。重建库后重跑 PGO 前,先 `cp outputs/{数据集}/db/trajectory.npz outputs/{数据集}/db/trajectory_orig.npz`,让 PGO 基于当前数据。
3. **web 改了不生效**:浏览器对同名 `data.js` 强缓存,硬刷新 `Ctrl+Shift+R`。
4. **DB 节点无法设起终点**:确认 `export_web.py` 的"轨迹走廊强制可通行"已生效(日志打印"覆盖虚构障碍 N 格"),DB 可达率应为全部节点。
5. **回环阈值取舍**:近距离回环靠 `--dmax`(太大会把矩形对边误判成回环、房间被挤扁),远距离大漂移真闭环靠 `--simfar`(太大漏闭环、太小引入对边误连)。**单程无重访的数据本就无闭环可修**,尺度漂移只能靠数据质量 / VGGT 改善。
6. **占据图障碍偏多**:位姿对齐后噪声点重合会形成虚假障碍,`export_web` 的 `grid_robust(min_hits=6)` 已配套滤噪,可按需调 `min_hits`。
7. **室外水平尺度偏小**(如 floor1):VGGT 单目室外水平运动尺度系统偏小,用 `imu_scale.py` 估全局尺度因子 k,再 `apply_scale.py <k>` 把 xy 乘 k(z 不变,幂等),最后 `export_web.py` 重新导出。
8. **穿玻璃会议室**:玻璃透明,被动视觉重建不出障碍点、被当作 free,A* 可能穿过(物理不通);根治需更稠密重建 / 主动深度传感。
9. **L40→GitHub push 失败**:`https` 偶发 TLS 握手失败,`git config http.version HTTP/1.1` 后重试;需有写权限的 token。
